# M14 + M11 Design: Person Awards Name Fallback & Batch Keyword Discovery

## M14: Person Awards Name Fallback

### Problem

`get_person_awards` only accepts a TMDB person ID with no name fallback. When TMDB ID and IMDb ID resolution both fail in Wikidata, the tool throws an error. The `resolvePerson` helper already supports an optional `name` parameter for name-based fallback (used by `get_film_awards` crew resolution since M9), but `handleGetPersonAwards` never passes one.

### Change

1. Add optional `name` field to `GetPersonAwardsSchema`
2. Pass `name` to `resolvePerson()` in `handleGetPersonAwards`
3. Update tool description to mention name fallback

Three lines of production code. No new dependencies, no architectural changes.

### Response Shape

Unchanged. The `entity.resolvedVia` field already supports `"name_search"` as a value.

## M11: Batch Keyword Discovery

### Problem

Thematic discovery requires sequential single-keyword lookups before feeding IDs into `discover`. A user researching "masculinity AND toxic masculinity AND fraternity" must make three separate `search_keywords` calls.

### Change

Enhance `search_keywords` to accept either a single string or an array of strings for the `query` field.

- **Single string** (existing behavior): calls TMDB `/search/keyword` once, returns paginated result. `page` parameter applies.
- **Array of strings** (batch mode): makes parallel TMDB calls (one per query), returns results keyed by query term. `page` parameter is ignored (each query returns page 1).

### Schema

```typescript
query: z.union([
  z.string().min(1),
  z.array(z.string().min(1)).min(1)
])
```

### Response Shape

**Single mode** (unchanged):
```json
{
  "page": 1,
  "results": [{"id": 194226, "name": "masculinity"}],
  "total_pages": 1,
  "total_results": 1
}
```

**Batch mode** (new):
```json
{
  "masculinity": {
    "results": [{"id": 194226, "name": "masculinity"}],
    "total_results": 1
  },
  "fraternity": {
    "results": [{"id": 15926, "name": "fraternity"}],
    "total_results": 1
  }
}
```

Batch mode omits pagination fields (`page`, `total_pages`) since it always returns page 1. If a user needs deeper pages for a specific term, they use single-query mode.
